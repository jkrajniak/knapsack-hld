## Why `mcknap.c` is the canonical artefact

From the file header (L20-40):

> The current code is intended for performing extensive tests with
> randomly generated instances. […]
> The code is run by issuing the command
>
>   mcknap k n r type
>
> where k: number of classes,
>       n: number of items in each class,
>       r: range of coefficients,
>       type: 1=uncorr., 2=weakly corr., 3=strongly corr., 4=subset sum
>             5=zig-zag (sinha-zoltners), 6=0-1knapsack problem

Generator logic lives at L754-782 (weight/profit sampling rules per
correlation type) and L814 onward (test-class generation). RNG is
seeded via `srand48` at L1606, with the seed coming from the test loop
(`START=1, TESTS=100`, L59-60), making instance regeneration
deterministic given `(k, n, r, type, seed)`.

## Recommendation

**P-port.** Rationale:
1. The pinned instance pipeline in `knapsack-hld/instances/` already
   speaks Python and emits the `InstanceModel` schema the solvers
   consume. A Python port keeps everything in one toolchain.
2. The generator is ~70 lines of straightforward C; porting is
   bounded, reviewable work.
3. The port is deterministic given a seed, so the canonical "Pisinger
   1995 instance set" becomes `(k, n, r, type, seed) ∈ <published
   grid>` rather than a frozen tarball — which is what the literature
   does anyway.
4. We escape the portability quagmire of compiling 1993 C on macOS 26.

D-descope is the fastest path if R.7 is not load-bearing for the
revision response; this is the PI's call.
